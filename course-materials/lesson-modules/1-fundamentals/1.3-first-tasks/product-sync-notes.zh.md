=== 通知讨论 ===

我汇报通知重设计提案

现状很糟：
- 上季度 127 个与通知相关的工单
- 用户要么每天收 50+ 条，要么全关
- 42 种通知类型（太多！）
- 偏好仅有开/关（不够细）
- 同步发送 ⇒ API 慢（200-400ms，Mike 说的）
- 企业客户在售前抱怨

提案：3 个层级（urgent/important/fyi）
- urgent：立即（被指派、@提及、截止）
- important：可进摘要（评论、状态变更）
- fyi：仅主动订阅（点赞、关注、一般活动）

智能批量：相关通知合并
例：同一任务 5 条评论 ⇒ 1 条通知，而不是 5 条

摘要模式：每日或每日两次
紧急仍即时发送

时区感知：尊重工作时间（用户时区 9-6）
工作时间外排队，稍后送达

更好的内容：展示任务标题+预览、操作者、操作按钮

MIKE 的技术备注：
“要异步队列，同步拖垮我们”
- 迁到 redis（我们已有）
- 通知 worker 处理队列
- 实现 3-4 周

DB 变更：
“user_preferences 表是 42 个布尔列，无法扩展”
想改成 jsonb
展示了示例 schema（urgent/important/fyi + 渠道、摘要设置、静默时段）

迁移需 2 周，不能一键切换，要渐进
现有用户默认“全部 = important”

“异步会提升性能，通知是最慢的 API 操作”
