# Carl 的 PRD 模板

# 问题对齐

## 问题与机会

用1-2句话描述我们要解决的问题。我应该能够单独阅读这句话，并向其他人传达价值/风险。

- 为什么这对我们的客户和业务很重要？
- 你有什么证据或见解来支持这一点？
- 我们正在与哪些客户合作或听取哪些客户的意见，以更好地理解这个问题？
- 为什么解决这个问题很紧迫？为什么是现在？
- 为什么它很重要？

## **高层级方法**

描述我们可能如何解决这个问题的粗略形态。我应该能够眯起眼睛看到同样的形态。例如，如果问题是“新功能的可发现性”，那么这可能是“相关功能的通知中心”。

我们考虑了哪些替代方案？为什么我们选择了这个？

### **叙述**

**可选**：分享（假设的）故事来描绘客户今天的生活是什么样的。描述设计解决方案时要考虑的常见和边缘用例。

## **目标**

1. *描述高层级目标，最好按优先级排序，不要太多。*
2. *包括可衡量的（指标）和不可衡量的（感受）目标*
3. *保持简短精炼*
4. *包括护栏指标*

## **非目标**

1. *列出我们不打算解决的明确领域*
2. *解释为什么它们不是目标*
3. *这些与目标一样重要和清晰*

# 解决方案对齐

## 关键功能

记录在案的计划

1. *列出构成解决方案的功能*
2. *最好按优先级排序*
3. *把这想象成画出解决方案空间的周长*
4. *划定界限，以便团队可以专注于如何填充它*
5. *对于特别大的项目，链接到子文档以获取更多详细信息*
6. *挑战规模，看看是否可以独立发布较小的组件*

未来考虑

1. *可选地列出你留待以后处理的功能*
2. *这些可能会影响你现在的构建方式*

### **关键流程**

展示客户的端到端体验将是什么。这可以是文字叙述、流程图、截图或设计探索。这会因项目和团队而异。不要试图孤立地做这件事。与设计和工程部门合作完成。

这一部分随着时间的推移变得更加具体是很自然的。它可能开始时只是一些带注释的截图或故事。它可能会变成具有验收标准的非常详细的需求。适应你团队的运作方式。如果你有一个喜欢深入研究每个边缘情况的强大设计师，请依靠他们。如果你有喜欢记录每个场景的详细工程师，请深入研究验收标准。

这自然会随着时间而改变——没关系。当发生更改时，在 [变更日志](https://docs.google.com/document/d/1mEMDcHmtQ6twzNlpvF-9maNlAcezpWDtCnyIqWkODZs/edit#heading=h.n2tmu89lqb7n) 中记录它们并通知所有贡献者。

### **关键逻辑**

1. *列出指导设计和开发的规则*
2. *解决常见场景和边缘情况*
3. *写出这些通常比依靠设计来展示每一种排列组合更容易*
4. *包括非功能性需求*
