## 3.2.2: 战略与架构可视化

- **完成时间:** 20 分钟
- **先决条件:** 用户与产品可视化 (模块 3.2.1)

> **在 Claude Code 中开始此模块:** 运行 `/start-3-2-2` 开始互动体验。

## 概述

模块 3.2.2 教你为不同的利益相关者受众创建战略和架构视觉内容。CTO 想了解技术架构。CEO 想看优先级。董事会想看路线图。不同的受众，不同的视觉内容。

**核心要点:** 战略视觉内容将复杂的想法转化为即时的清晰度。一张正确的图表可以在几秒钟内完成 10 分钟解释无法完成的任务。


## 场景

你的 TaskFlow Mobile 推介落地了。现在你有后续会议：

1. **CTO 会议** - "它如何与我们现有的系统集成？"
2. **CEO 会议** - "我们首先构建什么，为什么？"
3. **董事会会议** - "3 个季度的路线图是什么？"

每一个都需要不同类型的视觉内容。


## 系统架构图

![系统架构图示例 - 技术风格](/images/nano-banana/system-diagram-technical.png)

### 它们是什么

系统架构图展示组件如何连接——服务、数据库、API、外部集成。它们回答“技术系统是如何工作的？”

### 何时使用

- 与工程团队的技术深度探讨
- CTO/工程副总裁演示
- 安全和合规审查
- 集成规划讨论

### 如何生成

**基本提示词:**
```
Create a system architecture diagram showing [system].
Components: [list components]. Show data flow between them.
Clean technical style, suitable for [audience]. 16:9 aspect ratio.
(创建一个展示 [系统] 的系统架构图。
组件: [列出组件]。展示它们之间的数据流。
干净的技术风格，适合 [受众]。16:9 纵横比。)
```

**示例 - TaskFlow Mobile:**
```
Create a system architecture diagram for TaskFlow Mobile.

Components:
- TaskFlow Mobile app (the new thing)
- TaskFlow Backend API (existing)
- PostgreSQL database
- Redis cache
- Firebase (push notifications)
- Mixpanel (analytics)

Show data flow arrows between components.
Clean technical style suitable for CTO review. 16:9 aspect ratio.
(为 TaskFlow Mobile 创建一个系统架构图。

组件:
- TaskFlow Mobile 应用 (新事物)
- TaskFlow 后端 API (现有)
- PostgreSQL 数据库
- Redis 缓存
- Firebase (推送通知)
- Mixpanel (分析)

展示组件之间的数据流箭头。
适合 CTO 审查的干净技术风格。16:9 纵横比。)
```

### 关键元素

| 元素 | 目的 |
|---------|---------|
| **方框/节点** | 代表服务、数据库、应用 |
| **箭头** | 展示数据流和关系 |
| **标签** | 标识每个组件 |
| **分组** | 聚集相关组件 |
| **颜色** | 区分类型 (新 vs 现有，内部 vs 外部) |

### 风格选项

| 风格 | 最适合 |
|-------|----------|
| **干净/极简** | 高管演示 |
| **技术/详细** | 工程审查 |
| **多彩/品牌化** | 外部文档 |
| **手绘** | 白板/研讨会感觉 |

### 专业提示

- **突出新内容:** 使用颜色或边框区分新组件
- **展示边界:** 对内部与外部进行分组，或按团队所有权分组
- **保持可读性:** 每张图最多 5-10 个组件；如果需要则拆分
- **添加图例:** 如果有意义地使用了颜色或形状


## 优先级矩阵

![优先级矩阵示例 - 企业风格](/images/nano-banana/matrix-corporate.png)

### 它们是什么

2x2 矩阵在两个维度上绘制项目——通常是影响 vs 努力，或价值 vs 复杂性。它们使优先级决策可视化。

### 何时使用

- 功能优先级讨论
- 高管战略会议
- 规划会议
- 利益相关者对齐

### 经典的 2x2

最常见的: **影响 vs 努力 (Impact vs Effort)**

```
High Impact │  Major      │  Quick
            │  Projects   │  Wins
            │ (重大项目)  │ (速赢)
────────────┼─────────────┼──────────
Low Impact  │  Avoid      │  Fill-ins
            │ (避免)      │ (填充)
            │             │
            └─────────────┴──────────
             High Effort   Low Effort
```

| 象限 | 含义 | 行动 |
|----------|---------|--------|
| **速赢 (Quick Wins)** | 高影响，低努力 | 优先做 |
| **重大项目 (Major Projects)** | 高影响，高努力 | 仔细规划 |
| **填充 (Fill-ins)** | 低影响，低努力 | 有时间时做 |
| **避免 (Avoid)** | 低影响，高努力 | 不做 |

### 如何生成

**基本提示词:**
```
Create a 2x2 prioritization matrix.
X-axis: [dimension 1] (Low → High)
Y-axis: [dimension 2] (Low → High)

Quadrant labels: [list quadrants]

Plot these items:
- [Item 1]: [quadrant]
- [Item 2]: [quadrant]
...

Clean corporate style for executive presentation. 16:9 or 1:1.
(创建一个 2x2 优先级矩阵。
X 轴: [维度 1] (低 → 高)
Y 轴: [维度 2] (低 → 高)

象限标签: [列出象限]

绘制这些项目:
- [项目 1]: [象限]
- [项目 2]: [象限]
...

适合高管演示的干净企业风格。16:9 或 1:1。)
```

**示例 - TaskFlow Mobile 功能:**
```
Create a 2x2 prioritization matrix for TaskFlow Mobile features.

X-axis: Effort (Low → High)
Y-axis: Impact (Low → High)

Quadrants: Quick Wins (top-right), Major Projects (top-left),
Fill-ins (bottom-right), Avoid (bottom-left)

Plot these features:
Quick Wins:
- Push notifications for task updates
- Offline task viewing

Major Projects:
- Real-time sync across devices
- Team chat integration

Fill-ins:
- Custom themes
- Widget support

Avoid:
- Voice command support
- AR task visualization

Clean corporate style for CEO presentation. Square format.
(为 TaskFlow Mobile 功能创建一个 2x2 优先级矩阵。

X 轴: 努力 (低 → 高)
Y 轴: 影响 (低 → 高)

象限: 速赢 (右上), 重大项目 (左上),
填充 (右下), 避免 (左下)

绘制这些功能:
速赢:
- 任务更新的推送通知
- 离线任务查看

重大项目:
- 跨设备实时同步
- 团队聊天集成

填充:
- 自定义主题
- 小部件支持

避免:
- 语音命令支持
- AR 任务可视化

适合 CEO 演示的干净企业风格。方形格式。)
```

### 其他矩阵类型

| 矩阵 | 维度 | 用例 |
|--------|------------|----------|
| **艾森豪威尔** | 紧急 × 重要 | 任务管理 |
| **风险** | 概率 × 影响 | 风险评估 |
| **RICE** | 覆盖面 × 影响 × 信心 × 努力 | 功能评分 |
| **波士顿 (BCG)** | 市场增长 × 市场份额 | 投资组合战略 |

### 专业提示

- **明确象限标签:** 不要假设 Gemini 知道你的术语
- **清晰绘制项目:** 列出每个项目属于哪里
- **保持平衡:** 每个象限 3-5 个项目看起来最好
- **使用你的风格库:** 保存你喜欢的矩阵风格以供重用


## 产品路线图

![产品路线图示例 - 时间轴风格](/images/nano-banana/sequence-timeline.png)

### 它们是什么

产品路线图展示随时间推移的计划工作——通常按季度或月份。它们回答“计划是什么，何时进行？”

### 何时使用

- 董事会演示
- 年度计划审查
- 团队对齐
- 外部利益相关者更新

### 路线图结构

| 结构 | 最适合 |
|-----------|----------|
| **时间轴 (Timeline)** | 清晰的时间顺序视图 |
| **泳道 (Swimlane)** | 多个并行轨道 |
| **现在/下一步/以后 (Now/Next/Later)** | 灵活，无具体日期 |
| **里程碑 (Milestone)** | 关键交付物聚焦 |

### 如何生成

**基本提示词:**
```
Create a product roadmap for [product].

Timeline: [time period]

[Period 1]: [Theme]
- [Item 1]
- [Item 2]

[Period 2]: [Theme]
- [Item 1]
- [Item 2]

...

Professional style for [audience]. 16:9 aspect ratio.
(为 [产品] 创建一个产品路线图。

时间轴: [时间段]

[时期 1]: [主题]
- [项目 1]
- [项目 2]

[时期 2]: [主题]
- [项目 1]
- [项目 2]

...

适合 [受众] 的专业风格。16:9 纵横比。)
```

**示例 - TaskFlow Mobile:**
```
Create a product roadmap for TaskFlow Mobile over 3 quarters.

Q1 - MVP Launch:
- Core task management
- Push notifications
- Offline viewing

Q2 - Team Features:
- Real-time sync
- Team chat integration
- Shared task lists

Q3 - Enterprise:
- Admin controls
- SSO integration
- Advanced analytics

Professional style suitable for board presentation.
Show clear progression from MVP to enterprise-ready.
16:9 aspect ratio.
(为 TaskFlow Mobile 创建 3 个季度的产品路线图。

Q1 - MVP 发布:
- 核心任务管理
- 推送通知
- 离线查看

Q2 - 团队功能:
- 实时同步
- 团队聊天集成
- 共享任务列表

Q3 - 企业级:
- 管理员控制
- SSO 集成
- 高级分析

适合董事会演示的专业风格。
展示从 MVP 到企业就绪的清晰进展。
16:9 纵横比。)
```

### 关键元素

| 元素 | 目的 |
|---------|---------|
| **时间标记** | 季度、月份或 Sprint |
| **主题** | 命名每个阶段 |
| **功能** | 具体交付物 |
| **里程碑** | 关键日期或成就 |
| **进度** | 当前状态 (可选) |

### 专业提示

- **讲故事:** 每个阶段应逻辑地建立在前一个阶段之上
- **使用主题:** "MVP" → "增长" → "规模化" 比功能列表更清晰
- **不要过度细节化:** 路线图是战略性的，不是 Sprint 计划
- **匹配受众:** 董事会看主题；工程团队看功能


## 针对利益相关者的提示

### 给 CTO/工程团队

- **关注:** 集成点、数据流、技术依赖
- **包含:** 现有系统、API、数据库
- **风格:** 技术性但易读；不要让人不知所措
- **层级:** 架构层级，而非实现细节

### 给 CEO/领导层

- **关注:** 战略决策、优先级理由
- **包含:** 权衡、你不做什么
- **风格:** 干净、极简、高层级
- **层级:** 业务影响，而非技术细节

### 给董事会

- **关注:** 时间表、里程碑、战略方向
- **包含:** 阶段、关键交付物、业务成果
- **风格:** 专业、抛光、自信
- **层级:** 季度，而非每周


## 使用你的风格库

战略视觉内容从一致的风格中受益匪浅。

**保存你的常用风格:**
- "Executive Architecture" (高管架构) - 干净的技术图表
- "Corporate Matrix" (企业矩阵) - 专业的 2x2
- "Board Roadmap" (董事会路线图) - 抛光的时间轴视觉

**然后重用:**
```
Generate system architecture using style #34
Generate prioritization matrix using style #5
Generate roadmap using style #88
```

跨利益相关者材料的一致性看起来很专业。

## 最佳实践

### 要做:
- **根据受众匹配风格** - 工程师用技术风，高管用简洁风
- **讲一个清晰的故事** - 每个视觉内容应有一个主要信息
- **使用一致的风格** - 利用你的风格库
- **保持简单** - 复杂性扼杀理解
- **迭代** - 初稿很少是最终稿

### 不要:
- **不要过度复杂化** - 每张图最多 5-10 个元素
- **不要混合隐喻** - 选择一种视觉方法并坚持下去
- **不要省略标签** - 一切都应清楚标识
- **不要随意使用行话** - 匹配受众的词汇


## 故障排除

### 架构图太杂乱

- 仅保留核心组件
- 拆分为多个图表 (前端、后端、集成)
- 要求 "high-level overview, not detailed architecture" (高层概览，而非详细架构)

### 矩阵项目重叠

- 减少每个象限的项目 (3-5 个是理想的)
- 要求 "items clearly positioned, not overlapping" (项目清晰定位，不重叠)
- 使用更大的格式 (16:9 而不是 1:1)

### 路线图未显示进展

- 明确描述叙事: "MVP → Growth → Scale"
- 使用视觉进展 (从左到右，从小到大)
- 要求 "show clear evolution from phase to phase" (展示从阶段到阶段的清晰演变)
