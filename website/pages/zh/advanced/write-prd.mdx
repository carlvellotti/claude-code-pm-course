## 2.1: 撰写 PRD

 - **完成时间：** 45-60 分钟
 - **先决条件：** 模块 1.4 (代理), 模块 1.5 (自定义子代理)


> **在 Claude Code 中开始此模块：** 运行 `/start-2-1` 以启动交互式体验。

## 概览

模块 2.1 教您通过使用 AI 作为思维伙伴来更快地撰写更好的 PRD。您将通过 @-提及整合模板、公司背景和用户研究，并行生成多种战略方法，并在与团队分享之前获得多视角反馈。

**关键要点：** AI 最有价值的时候是它帮助您更好地思考，而不是当它为您完成所有思考时。最好的 PRD 结合了人类判断力与 AI 处理完整上下文、生成选项和提供多样化视角的能力。


## 传统方法 vs AI 伙伴关系方法

| 传统方法 | AI 伙伴关系方法 |
|---------------------|------------------------|
| 打开空白文档 | 通过 @-提及提供完整上下文 |
| 独自撰写 PRD | 使用苏格拉底式提问理清思路 |
| 与团队分享 | 生成多种战略方法 |
| 获得反馈 | 首先从子代理处获得反馈 |
| 修改 | 在任何人看到之前进行完善 |
| 重复 | 分享更强大的初稿 |

**区别：** 您永远不会从零开始，并且在第一次审查之前已经解决了潜在问题。


## 四种核心技术

### 1. 通过 @-提及提供完整上下文

一次性为 Claude 提供所有相关信息。

**@-提及什么：**
- **公司背景** - 产品战略、客户洞察、业务指标
- **用户研究** - 访谈记录、调查结果、痛点
- **模板** - 您首选的 PRD 结构
- **方法** - 像苏格拉底式提问这样的框架
- **相关文档** - 以前的 PRD、竞争分析、路线图

**示例：**
```
使用以下内容帮我为 [功能] 撰写 PRD：
@company-context.md - 产品和客户背景
@user-research/pain-points.md - 来自 20 次访谈的洞察
@prd-template.md - 我们的标准 PRD 结构
@methods/socratic-questioning.md - 清晰思维的框架
```

### 2. 用于清晰度的苏格拉底式提问

在写作之前使用结构化提问来磨练模糊的思维。

**关键问题类别：**

| 类别 | 问题 |
|----------|-----------|
| **问题清晰度** | 这解决了什么具体的用户痛点？<br/>谁最强烈地感受到这一点？<br/>不解决这个问题的成本是什么？ |
| **解决方案验证** | 为什么这是正确的解决方案？<br/>您考虑了什么替代方案？<br/>最简单的版本是什么？ |
| **成功标准** | 我们将如何衡量成功？<br/>什么将表明失败？<br/>我们要移动什么指标，移动多少？ |
| **约束** | 存在什么技术风险？<br/>我们不做 什么？<br/>如果我们只有一半的时间，我们会砍掉什么？ |
| **战略契合** | 为什么 现在 构建这个？<br/>这如何适应我们更广泛的战略？<br/>这如何影响竞争地位？ |

**专业提示：** 选择 3-5 个最相关的问题。质量优于数量。

### 3. 生成多种战略方法

让 Claude 为同一功能生成 3 种不同的战略方法。比较、选择最好的，或组合元素。

**示例：AI 语音聊天 + 待办事项列表功能**

| 方法 | 主要界面 | 用例 | 成功指标 |
|----------|------------------|----------|----------------|
| **聊天优先** | AI 对话 | 移动时快速捕捉 | 通过语音创建的任务 % |
| **列表优先** | 传统待办事项列表 | 现有工作流的免提更新 | 语音功能的 DAU |
| **平衡** | 两者权重相等 | 基于上下文灵活使用 | 用户满意度 + 采用率 |

**使用代理进行并行生成：**
```
启动 3 个代理并行生成 3 个 PRD 草稿：
- 代理 1: 聊天优先方法
- 代理 2: 列表优先方法
- 代理 3: 平衡方法
```

这展示了并行处理对于 PM 工作的力量。

### 4. 通过子代理进行多视角反馈

在与您的实际团队分享之前获得多样化的反馈。

**三个关键视角：**

| 子代理 | 关注领域 |
|-----------|-------------|
| **👨‍💻 工程师** | 技术可行性、实施复杂性、风险、依赖关系、时间线估算 |
| **💼 高管** | 商业价值、战略契合、资源理由、竞争定位、ROI |
| **👤 用户研究员** | 用户需求一致性、可用性问题、边缘情况、采用障碍、指标有效性 |

**示例：**
```
从所有三个子代理处获得审查并汇总反馈：
- @.claude/agents/engineer.md - 技术审查
- @.claude/agents/executive.md - 商业审查
- @.claude/agents/user-researcher.md - UX 审查
```


## 现实世界示例

**背景设置：**
```
我正在为智能会议摘要撰写 PRD。帮我使用以下内容思考这个问题：
@company-context.md
@user-research/calendar-pain-points.md
@prd-template.md
@methods/socratic-questioning.md
```

**理清思路：**
```
使用苏格拉底框架问我关于此功能的 5 个关键问题：
- 问题清晰度
- 为什么是这个解决方案
- 成功指标
- 技术约束
- 战略时机
```

**生成方法：**
```
生成 3 种战略方法：
1. 日历原生 (摘要出现在日历应用中)
2. 邮件优先 (摘要通过邮件发送)
3. 独立应用 (专用会议中心)

针对每一个：问题框架、解决方案、成功指标
```

**获得反馈：**
```
👨‍💻 工程师，审查 @smart-summaries-prd.md 的技术可行性。
然后 💼 高管，审查商业价值和战略契合。
然后 👤 用户研究员，审查用户需求一致性。
```


## 最佳实践

**做：**
- 从完整上下文开始 (@-提及公司文档、研究、模板)
- 使用苏格拉底式问题来理清模糊的思维
- 生成 2-3 种战略方法，不要只用第一个想法
- 在与真实团队分享之前获得子代理反馈
- 自己驱动流程 - AI 提供选项，你做决定

**不做：**
- 让 AI 为你写作 (你是合作伙伴，不是外包)
- 跳过思考 (问题的存在是为了磨练 你的 判断力)
- 接受初稿 (总是生成替代方案进行比较)
- 忽略子代理反馈 (通常能捕捉到重要问题)
- 孤立工作 (使用完整上下文)

**专业提示：**

| 提示 | 好处 |
|-----|---------|
| 创建方法库 | 适用于任何 PRD 的可重用框架 (Socratic, JTBD, RICE) |
| 维护模板库 | 适用于功能、重设计、实验的不同模板 |
| 保存好的代理提示词 | 重用能提供有价值反馈的角色 |
| 使用检查点 | 保存每个版本 (v1, v2, v3) 以参考替代方案 |
| 记录推理 | 记录 为什么 你选择这种方法而不是其他方法 |


## 故障排除

**AI 生成的 PRD 感觉很通用**

原因：上下文不足或 AI 自主性太强

解决方法：
- 提供更具体的公司上下文 (@-提及详细文档)
- 使用苏格拉底式提问注入 你的 思考
- 要求“帮我思考”而不是“帮我写 PRD”
- 添加来自研究的具体示例和用户语录

**子代理反馈停留在表面**

原因：代理角色不够具体，或 PRD 缺乏深度

解决方法：
- 用具体的关注点增强子代理角色 (API 速率限制、可扩展性)
- 为代理提供更多上下文 (@-提及技术/战略文档)
- 要求特定类型的反馈：“专注于技术风险”
- 确保您的 PRD 草稿有足够的细节供审查

**流程耗时太长**

原因：步骤太多，或没有使用并行化

解决方法：
- 您不需要每次都用所有 3 种方法 - 一种可能就足够了
- 使用代理并行生成草稿 (而不是顺序生成)
- 如果您已经很清楚，可以跳过苏格拉底式提问
- 并非每个 PRD 都需要子代理审查 - 用于高风险功能

**难以跟踪版本**

原因：文件名或版本管理不佳

解决方法：
- 清晰命名：`feature-prd-v1.md`, `feature-prd-v2-chat-first.md`, `feature-prd-final.md`
- 添加日期：`feature-prd-2025-01-15.md`
- 在 PRD 中保留变更日志部分
- 使用 git 提交来跟踪演变
- 将旧版本归档在 `drafts/` 文件夹中
